# 从开发者视角理解 BuckyOS 的分布式权限系统

> 说明  
> 本文根据一段开发口述录音整理而成，目标是建立**开发者视角**下的理解框架，而不是逐行解释现有实现。  
> 个别英文名在录音中存在转写误差，正文尽量按现有项目习惯统一，例如：`node-daemon`、`buckycli`、`verify-hub`、`node-gateway`、`BuckyOS App`。  
> 文中“鉴权”既包含 **JWT 校验**，也包含后续 **RBAC 权限判断**。

## 1. 一句话总览

BuckyOS 的“权限系统”并不是一个孤立模块，而是**一组底层服务正确使用鉴权组件之后的综合结果**。

换句话说：

- 如果底层服务没有在正确的位置做鉴权，再完备的权限模型也不会真正生效。
- 越底层的组件，越关注“**我该如何校验请求是否合法**”。
- 越上层的组件，越关注“**我该如何拿到一个能用的授权**”。

这也是为什么很多人会觉得这套系统复杂：它不是“只有一个权限中心”，而是**鉴权点分布在多个层级**。

---

## 2. 权限系统真正落地在哪里

从开发者视角看，权限真正落地的地方主要有两类：

1. **服务自身收到请求之后的显式鉴权**
2. **`node-gateway` 在 HTTP 入口处做的隐式鉴权**

前者是大多数业务代码最常接触的地方；后者则是很多“明明后端没报错，但请求就是过不去”的来源。

因此理解权限系统时，建议按下面的顺序建立心智模型：

1. 先理解 **JWT / Token 中到底携带了什么身份信息**
2. 再理解 **服务收到请求后如何做两阶段鉴权**
3. 最后理解 **Gateway 为什么会在请求抵达服务前就把它拦掉**

---

## 3. 权限的最小载体：JWT Token

当前系统里，权限相关的会话信息本质上都基于 JWT。

录音里把它叫做 `session token`，但更准确地说，日常参与鉴权的是 `access token`。  
系统整体上把几乎所有这类凭证都统一成 JWT 来表达。

### 3.1 JWT 里最关键的字段

从理解系统行为的角度，最重要的是下面几项：

| 字段 | 含义 | 开发者应如何理解 |
| --- | --- | --- |
| `userID` / `sub` | 当前操作的“用户主体” | 谁在做这件事 |
| `appID` | 当前操作的“应用主体” | 这个请求是在哪个应用/运行时上下文里发起的 |
| `iss` | 签发者 | 这个 Token 是谁发的 |
| `exp` | 过期时间 | 过期即无效，主要是安全性约束 |
| `aud` / `audience` | 受众 | 这个 Token 是不是只能给某个特定服务使用 |
| `kid` | Key ID | 可选，用于辅助定位签名 key |

其中最容易让人误解的是 `appID`。

### 3.2 为什么 `appID` 很重要

在 BuckyOS 里，`appID` 不只是“客户端名称”，它本质上代表：

- 这个请求从哪个软件上下文发起；
- 换成更底层的话说，是“**哪个用户态执行体在发起请求**”。

所以可以把它理解成“第二主体”。

也正因为此，BuckyOS 的权限判断**不是只看 userID**，而是通常会同时看：

- **用户主体**
- **应用主体**

这也是后面“两次 RBAC 判断”的来源。

### 3.3 `audience` 在当前系统里的实际用法

从录音里的描述看，`audience` 在当前系统中的语义非常直接：

- **不填 `audience`**：通常表示这是一个更通用的 Token，可以被多个系统服务使用；
- **填写 `audience`**：表示这是一个受限 Token，只应该被特定目标接受。

在当前实现里，**受限 `audience` 的 access token 还比较少见**。  
更常见的是 **refresh token 会明确限制 `audience = verify-hub`**，因为它的唯一目的就是去换新的 access token，而不是直接访问普通业务服务。

这也是理解 token 使用场景时的一个经验法则：

- 拿到一个 access token，通常先按“服务间通用”理解；
- 拿到一个 refresh token，则优先按“只能去 verify-hub 换票”理解。

## 4. BuckyOS 对 JWT 的工程化约定

### 4.1 历史原因：内核引导阶段曾把 JWT 塞进 DNS TXT

系统最早在引导阶段，需要把一部分 boot 信息放进 DNS TXT record。  
但 DNS TXT 的长度有限，因此系统并没有完全按“标准 JWT 最宽松、最完整”的写法来用，而是做过一轮**最小化裁剪**。

结果就是：

- 某些标准字段在 BuckyOS 里允许省略；
- 省略后会使用系统约定的默认值；
- 因此有时看起来“不像一个标准 JWT”，但在系统语义里它仍然是可推导、可验证的。

### 4.2 常见默认值

录音里给出的默认语义大致如下：

| 字段未填写时 | 默认语义 |
| --- | --- |
| `appID` 为空 | 视为“内核” |
| `userID` 为空 | 视为 `root` 用户 |
| `iss` 为空 | 视为当前验证环境里的“根信任 key” |
| `kid` 为空 | 很常见，通常不填 |

补充两点：

1. `iss` 的默认语义依赖环境。可以把它理解为“使用当前环境里默认的根信任签发者”。
2. `kid` 有时会出现在 header 里，但系统里它并不总是严格等同于某个标准 JWT 语义下的子标识；在动态环境中，`kid` 和 `iss` 都可能出现以 `$` 开头的变量化写法。

### 4.3 这会带来什么工程上的观感问题

很多开发者第一次看系统里的 JWT helper 时会觉得“这怎么像是不合法的 JWT”。  
问题不在于系统不理解 JWT，而在于：

- BuckyOS **保留了标准 JWT 的骨架**
- 但为了适配内核引导、DNS 长度限制、动态环境等历史包袱
- 在字段可选性和默认值推导上做了不少工程化处理

所以理解时，**先按标准 JWT 看，再叠加 BuckyOS 的默认值规则**，通常最容易看懂。

## 5. 标准鉴权流程：先验 Token，再做双主体 RBAC

服务端最核心的显式鉴权流程可以简化为：

```text
收到请求
  -> 校验 JWT 是否有效
  -> 提取 userID / appID
  -> 根据请求参数构造资源路径
  -> 用 userID 做一次 RBAC 判断
  -> 用 appID 再做一次 RBAC 判断
  -> 两次都通过，才允许访问资源
```

### 5.1 第一阶段：JWT 本身是否合法

这一步主要判断：

- 签名是否有效
- `iss` 是否可信
- Token 是否过期
- `audience` 是否匹配（如果该 Token 是受限受众）

这一阶段回答的是：**“这张票是真的吗？”**

### 5.2 第二阶段：RBAC 权限是否允许

JWT 验证通过之后，系统会把 `userID` 和 `appID` 都拿出来参与权限判断。

录音里的核心结论是：

> 在 99.9% 的情况下，系统默认采用**交集权限**。

也就是说：

- 用户主体有权限，不代表请求就能过；
- 应用主体有权限，也不代表请求就能过；
- **两者都得有权限**，这个请求才能真正通过。

所以可以把 `appID` 看成“第二个 subject”，只是它被系统单独提出来做了工程化简化。

### 5.3 RBAC 里真正关键的是“资源路径怎么拼”

RBAC 不是对“请求”本身做判断，而是对“某个主体能否访问某个资源”做判断。

因此一个服务最关键的职责之一是：

- 从当前请求的参数中
- 提取出足够的信息
- 拼成一个稳定的“资源路径”
- 再拿这个路径去查 RBAC

通常这个资源会是某种路径表达式，例如：

- 某个用户的设置
- 某个 APP 下的配置
- 某个数据对象
- 某个服务下的某个子资源

### 5.4 有些服务可以只转发，不在本地做最终鉴权

如果某个服务本身只是一个中转层，接收到请求后会把它拆成多个子请求继续发往下游服务，那么它**不一定非要在本地完成最终资源鉴权**。

这种情况下，它可以：

- 只保证调用链上的 Token 合法；
- 然后把请求继续往下传；
- 最终由真正拥有资源的底层服务完成 RBAC 判断。

这也是“权限落点分布式”的一个体现。

## 6. 一个容易犯错的原则：调用语义和目标语义必须分开

这部分是整段录音里非常重要的设计原则。

很多 API 设计者会想当然地认为：

- JWT 里已经有 `userID` 和 `appID`
- 那我的接口参数里是不是就不用再写 `userID` / `appID` 了？

答案是：**不能这样偷懒。**

### 6.1 什么时候必须把 `userID` / `appID` 明确写在参数里

只要这两个字段参与了**资源路径构造**，它们就必须显式出现在参数里。

原因是：

- JWT 里的主体，表示的是“**谁在调用**”
- 参数里的主体，表示的是“**访问目标是谁**”

这两者不一定相同。

### 6.2 一个典型例子：读取某个用户某个 APP 的 setting

假设有一个接口叫 `getSettings`，语义是：

- 获取某个用户在某个 APP 下的设置

这时就不能偷懒说：

- “我直接从 JWT 里拿当前 userID / appID，拼成资源路径就好了”

因为系统最终一定会面临这样的问题：

- Bob 能不能访问 Alice 的 settings？
- 一个管理型 APP 能不能代替用户读取某个目标用户的资源？
- 共享服务能不能让多个用户访问同一应用上下文？

所以只要资源路径的目标身份可能与调用者身份不同，**目标身份必须显式建模**。  
不能用“JWT 里的主语”直接替代“资源的目标语义”。

## 7. 服务自己的运行时身份：不是只有“收到请求”这一种场景

很多系统服务不只是被动收请求，它们自己还会做后台任务。

例如：

- 定时任务
- 后台同步
- 任务状态更新
- 主动访问其他内核服务

这时就会出现第二类身份：**服务自己的运行时身份**。

### 7.1 系统内部服务大多通过 SDK 调服务

录音里提到，BuckyOS 内部服务在访问系统服务时，大多数都通过 SDK，而 SDK 的核心本质上就是一个 API Client。

从生命周期看，服务自己的运行时一般分成两步：

1. **初始化环境**
2. **登录（login）并获得自己的 Token**

登录之后，服务会持有自己的一组：

- `access token`
- `refresh token`

它们用于服务后续访问其他系统服务。

### 7.2 后台服务的 `appID` 和 `userID` 怎么填

这取决于服务类型。

#### 场景 A：系统级后台服务

如果是系统服务在后台运行：

- `appID`：就是它自己
- `userID`：当前设备

这是分布式系统视角下的一种重要约定：  
**后台系统服务的“用户”常常不是人，而是当前设备。**

也就是说，此时的语义更接近：

> 某台设备通过某个系统应用，在使用某种系统资源。

#### 场景 B：用户安装的 APP 服务

如果是一个明确属于某个用户的 APP 服务，那么它后台运行时的 `userID` 就更自然地落在**该用户本身**。

### 7.3 为什么服务启动后还要再 login 一次

服务初始启动时，往往先拿到的是**设备签发**的 Token。  
但这类 Token 的可识别范围有限，很多其他服务并不直接认它。

因此服务通常还要经过一次 login，把它在 `verify-hub` 上完成校验和刷新，换成整个系统里更广泛可识别的 Token。

可以粗略理解为：

- 启动阶段：先拿到“本地可用”的身份
- 登录之后：换成“全系统都更容易识别”的身份

### 7.4 这套设计带来的一个好处

在常见情况下，一台设备上通常只需要重点持有两套根信任公钥：

- 当前设备自己的公钥
- `verify-hub` 的公钥

这样可以避免每台设备都去持有整个分布式系统里所有其他设备的公钥。

## 8. 收到用户请求后，到底该用谁的 Token？

这是业务开发里最容易写错的一块。

因为一个服务在处理“用户驱动的请求”时，手里往往会同时有两套身份：

1. **服务自己平时后台运行时的 Token**
2. **来自用户请求链路里的 Token**

### 8.1 为什么这会让人困惑

在很多简单场景里，这两张票看起来几乎一样。

比如：

- 同一个用户
- 同一个 `appID`
- 只是签发者不同

那么从权限效果看，二者往往都能用。

这会让开发者误以为“随便拿一个都行”。

### 8.2 什么时候必须严格区分

一旦出现共享 APP 或多用户使用同一服务的场景，就必须严格区分。

例如：

- 一个共享 APP 允许多个用户同时使用；
- 一个长任务是用户发起的，但后续在后台继续运行；
- 一个服务需要代用户访问下游组件。

这时如果混用 Token，风险主要有两类：

1. **权限扩散**  
   本来应该代表“某个具体用户”的操作，被错误地放大成“APP 自己的泛权限”。

2. **权限下降或用户污染**  
   本来应该允许的操作做不了，或者多用户之间的身份上下文被混淆。

### 8.3 长任务尤其麻烦

当前 access token 的有效期比较短（录音中给出的量级是 15 分钟）。  
因此如果一个任务生命周期很长，就不能简单地把“收到请求时的 access token”一直原样带到底。

这时服务通常需要认真设计：

- 是否保存调用者上下文；
- 是否在合适时机做 token 转换；
- 下游调用到底使用“用户请求 token”还是“服务自己的 token”。

录音里的判断是：**这里目前没有一条放之四海而皆准的通用规则。**

也就是说，这部分更像是**应用架构设计问题**，而不是单纯的内核规则问题。

### 8.4 现阶段的系统边界

虽然这里容易写错，但从系统爆炸半径来看，风险仍然是相对可控的。

因为从系统角度看，一个服务就算“用错了自己的 token”，它能达到的权限上限，通常也不会超过：

- 它所属用户
- 加上它自己的 `appID`
- 在当前系统里被授权的那部分权限上界

但对于“可多用户同时登录”的共享 APP，系统仍然倾向于只允许**足够可信的发行者**使用这种模式。

## 9. 高权限身份：owner、root、sudo

这一部分在口述里有一定历史包袱，因此建议按“高权限来源”来理解，而不是强行只套进一个概念。

### 9.1 system owner：系统初始化时的最高域外权力

`system owner` 可以理解为：

- 创建系统/Zone 时留下的那个最高信任身份
- 它的公钥会被系统记录
- 用于系统根拓扑、DNS 记录等关键初始化信息的判定

它的重要性在于：

- 系统必须知道“谁有资格定义这个系统最基础的结构”
- 否则任何人都可以伪造拓扑信息，系统会被拖进错误的集群状态

因此在权限层级上，`system owner` 属于非常高的一档，尤其是在**域外配置 / DNS / 根拓扑**这类事情上。

### 9.2 root：系统内建的特殊用户

系统里还存在一个固定存在的 `root` 用户。

录音里强调了两点：

- `root` 是系统内建的、长期存在的；
- 当 `userID = root` 时，`iss` 也必须是它自己，不能由别的主体替它签发。

可以把它理解为系统内部的一种“绝对高权限用户”。

### 9.3 普通用户的 sudo 身份

除正常的 SSO 登录身份外，某些管理员类型用户还会有一种“sudo 身份”。

它的语义大致是：

- 在域内权限上非常高；
- 除了 DNS 这类域外能力，以及极少数真正 root-only 的系统安全区外；
- 在大多数管理员操作上，它与更高权限身份接近等价。

例如一些高风险操作会要求 sudo 身份：

- 重型服务卸载
- 高危配置修改
- 其他管理员级行为

### 9.4 当前 sudo 体验为什么比较“重”

录音里提到，当前某些高权限操作仍然走一种比较重的流程：

- 用户需要显式提供自己的私钥介质；
- 系统内部不会保存这把最高权限私钥；
- 借助 `buckycli` 之类工具，可以构造出高权限 JWT 进入系统执行操作。

这套机制的优点是安全边界清晰，缺点是使用成本高。

从口述判断，后续系统很可能会把**非 owner 的 sudo 流程**进一步收敛到 `verify-hub` 上，而不是始终要求用户显式掏私钥。

## 10. 三类最常见的开发者运行时

从日常开发看，最常碰到的其实是三种运行时：

1. Web 前端
2. CLI / Agent 工具
3. BuckyOS App 运行时

---

### 10.1 Web 前端：最纯粹、最好理解

对于 Web 页面来说，权限模型反而相对简单。

典型流程是：

1. 初始化运行时环境
2. 调 `getAccountInfo`
3. 如果拿到了账户信息，说明当前页面已有登录态
4. 如果返回空，则走 `login`
5. 登录回来后，继续调用各种 service client 获取数据并填充 UI
6. 退出时调用 `logout`

Web 运行时的关键特点：

- 每个页面 / tab 的登录态是相对独立的；
- 浏览器场景下，一个 tab 可以是用户 A，另一个 tab 可以是用户 B；
- SDK 会在发 RPC 之前自动带上 access token；
- access token 快过期时，会自动尝试 refresh。

所以对 Web 开发者来说，最核心的心智模型基本就是：

- `getAccountInfo`
- `login`
- `logout`
- 然后正常调各种 Client

### 10.2 CLI / Agent 工具：权限边界还在演进

录音里提到，当前很多 Agent 工具的实现骨架是从 `buckycli` 演化出来的，但具体应该如何建模其 `appID`，其实还在思考中。

这里存在两种设计方向：

#### 方向 A：工具共享 Agent 的 `appID`

这样做的含义是：

- 工具只是 Agent 的能力延伸；
- 工具的权限基本继承 Agent 的权限边界。

#### 方向 B：工具自己拥有独立 `appID`

这样做的含义是：

- 即使多个 Agent 都使用同一个工具；
- 这个工具本身也仍然有自己独立的权限边界；
- 不会因为挂到某个“万能 Agent”下面，就天然变成危险工具。

这本质上是在讨论：

> 权限控制边界应该落在“Agent”上，还是落在“Tool”上？

录音没有给出最终结论，但已经明确指出：  
如果所有工具都直接继承一个过大的 Agent 权限，风险会偏高。

### 10.3 BuckyOS App：未来会比浏览器环境更强

还有一类规划中的运行时，是把今天浏览器里的 WebUI 搬进 `BuckyOS App` 里运行。

这个环境的关键差异是：

- 它运行在可信设备上；
- 本地运行时除了可以从 `verify-hub` 拿身份外，也能利用设备级身份；
- 整个 App 容器可以统一管理多个页面的登录态；
- 退出、切换身份时，可以由 App 统一控制，而不是像浏览器那样每个页面各自为政。

对 UI 开发者来说，当前的原则很明确：

1. 今天能在浏览器跑的 WebUI，将来搬进 `BuckyOS App` 后一定还能跑；
2. SDK 层面只会**加能力**，不会**减能力**；
3. 如果开发者愿意识别不同 runtime，还可以进一步利用本地能力，例如：
   - 更强的数据能力
   - 更好的离线能力
   - 更少受限于浏览器容器

## 11. `node-gateway` 上的隐式鉴权：另一半复杂度的来源

到这里为止，我们讲的都是“服务自己显式做鉴权”。  
但系统里还有一类非常重要、也非常容易让人困惑的鉴权：**Gateway 鉴权**。

这类鉴权是 HTTP 协议专用的，而且位置很深，发生在网关层。

### 11.1 为什么 Gateway 会参与权限控制

在 BuckyOS 里，大多数 HTTP 流量并不是直接打到某个具体服务进程的裸端口。

更常见的做法是：

- 先打到本机 `127.0.0.1:3180`
- 外部 `80/443` 进入的 HTTP/HTTPS 流量，最终也会转发到这里
- 由 `node-gateway` 根据 `host`、`path`、调度信息等
- 再把请求路由到正确的设备与服务进程

也就是说：

- `3180` 是整个集群视角下的统一入口；
- 它既处在流量入口，也处在流量出口；
- 跨设备访问，通常也仍然是先连本机 `3180`，再由它帮你转发。

既然它是统一入口，权限控制自然也会在这里发生。

### 11.2 Gateway 的鉴权视角与服务端不同

Gateway 不会像业务服务那样深入理解具体资源模型。  
它更擅长做的是：

- 研究流量来源
- 判断来源是否有资格打到目标位置
- 决定是放行、重定向、拒绝还是改写路由

所以它的视角更接近：

> “这个请求从哪来？能不能把流量送到那里去？”

而不是：

> “这个用户能不能操作数据库里的第 N 条记录？”

录音里还专门强调了一点：`node-gateway` 被刻意设计成一个**相对通用的可编程转发器**，它有自己的脚本/DSL 体系，尽量不把太多 BuckyOS 特有逻辑硬塞进去。  
这么做的好处是工程边界更清晰，坏处就是它更适合做“流量级鉴权”，而不适合替代业务服务去做完整 RBAC。

### 11.3 Gateway 通常只看得到 `appID`

录音里一个非常关键的点是：

- 在 Gateway 的视角里，最核心的是**来源 APPID**
- 它通常看不到业务语义下的 `userID`

因此它更像是在做：

- APP 到 Service 的流量级许可
- APP 到 APP 静态资源 / 页面资源的访问控制
- SSO 登录态检查

而不是真正代替底层服务去做精细 RBAC。

### 11.4 它为什么还能解决跨域问题

BuckyOS 的一个很重要的工程技巧是：

- 不是把服务暴露成一堆跨域端口；
- 而是把服务能力通过 URL 路径投影到 APP 自己的域名下。

例如在浏览器看来，一个 APP 的域名下会存在一组类似内部 API 路径的入口。  
Gateway 根据路径规则识别后，再把流量 upstream 到真正的目标服务。

效果就是：

- 浏览器看来是同域；
- 因而 99% 的情况下不需要专门处理跨域；
- 也不需要业务开发者自己去折腾复杂的 CORS 配置。

### 11.5 如果 APP 无权访问某个服务，会发生什么

很多人第一反应是“后端会报 500”。

但在 BuckyOS 的 Gateway 模型下，更可能发生的是：

- 这个路径在 Gateway 看来根本就不应该存在；
- 请求还没到后端，就已经被 Gateway 拒绝了；
- 因而你在服务端日志里甚至看不到这次请求。

这也是排查问题时最容易误判的地方。

## 12. 静态 Web APP 也会被鉴权

BuckyOS 不仅给“动态服务”做权限控制，也给“纯静态页面 APP”做权限控制。

### 12.1 静态 APP 的两种模式

一个 APP 大致可以有两种发布模式：

1. **公开模式**
2. **标准模式**

其中“标准模式”意味着：

- 这个 APP 默认要求用户已完成 SSO 登录；
- 未登录用户不能直接访问。

### 12.2 登录态是谁检查的

这里不是业务代码检查，而是 **Gateway 检查 HTTP Cookie**。

具体来说：

- 标准模式 APP 的请求必须带上相应的登录 Cookie；
- 如果没有，Gateway 会直接把请求改成 `302`，跳到 SSO；
- 登录回来之后，再继续访问原 APP。

所以对纯静态页面来说，系统相当于提供了：

- 不写一行鉴权代码
- 但仍然自动受登录态保护

这对开发体验很好，但它也会带来一个副作用：

> 因为太透明了，开发者很容易忘记“其实请求在网关层已经被改写或拦截过”。

## 13. 系统服务自带 UI 时，为什么还要配 APP 信息

系统服务开发者经常会疑惑：

- “我是一个内核服务，权限已经很高了”
- “为什么我的 UI 还得配 APP 信息 / 域名相关配置？”

原因很简单：

在 Gateway 看来，**凡是通过域名和 HTTP 访问的页面，本质上都得先被识别成一个 APP**。

如果服务没有把这层信息配置出来，那么：

- Gateway 无法从域名侧识别它；
- 它也就无法把请求归类到正确的 APP / Service 访问规则里；
- 结果就是页面路由、SSO 回调、自动鉴权都可能不正常。

因此哪怕是系统服务自带 UI，也往往需要：

- 配自己的 `appID`
- 配对应的 APP 信息
- 配与域名/路由相关的映射

某种意义上讲：

> 在 Gateway 的世界里，能被用户访问到的 UI，首先都必须“像一个 APP”。

当前系统里，系统域名本身也存在一层特殊照顾：  
像 `sys.<zone>` 这一类系统域名，以及 SSO 相关路径，通常会在 Gateway 规则里被额外识别和接管。  
这也是为什么很多“系统服务 UI 明明跑起来了，但网页登录流程不对”的问题，最终要回到 Gateway 规则和 APP 信息配置上来查。

## 14. 从开发者角度排查权限问题，应该先看什么

如果把这套系统压缩成一个实用排查清单，建议按下面顺序看：

### 14.1 先判断：问题发生在 Gateway，还是服务内部

最先问自己两个问题：

1. 请求有没有真的到达服务？
2. 如果没到，是否在 Gateway 就被拒绝、重定向或改写了？

### 14.2 再看 JWT 本身

重点看：

- `userID`
- `appID`
- `iss`
- `audience`
- 是否过期
- 哪些字段是显式填写的，哪些是靠默认值补出来的

### 14.3 再看资源路径是怎么拼的

这是 RBAC 能不能正确工作的关键。

要确认：

- 当前接口是“按调用者语义”还是“按目标对象语义”工作；
- 资源路径用到的 `userID` / `appID` 是来自参数，还是来自 JWT；
- 这里有没有把“调用者”和“目标对象”混为一谈。

### 14.4 再看是用哪一张 Token 在访问下游

尤其是服务同时持有：

- 自己后台运行时的 token
- 用户请求链路里的 token

要确认当前代码选的是哪一个，以及为什么。

### 14.5 如果是 Web / UI 问题，再查域名与 APP 信息

重点检查：

- APP 是否配置了对应域名
- Gateway 是否能把当前请求识别成某个 APP
- 静态 APP 是不是标准模式
- SSO 回调是不是被 Gateway 正确接住了

## 15. 这套系统真正想表达的几个原则

最后把整段录音浓缩成几条原则：

### 原则 1：权限不是单点能力，而是调用链正确性的结果

权限系统是否有效，取决于每一层是否在正确的位置用了正确的鉴权组件。

### 原则 2：`userID` 和 `appID` 都是主体

BuckyOS 的权限判断不是只看用户。  
应用本身也是主体，而且在绝大多数场景下采用交集权限。

### 原则 3：调用者身份不等于目标资源身份

凡是会影响资源定位的身份信息，都应该在接口层显式表达，而不是偷懒从 JWT 里直接推。

### 原则 4：服务收到请求后，不一定总该用“自己的 token”

特别是共享 APP、多用户场景、长任务场景，必须认真区分“服务身份”和“调用者身份”。

### 原则 5：Gateway 的拒绝不是后端错误

如果一个请求根本没到服务，就不要去后端里找业务 Bug。  
先查 Gateway 规则、来源识别、域名映射和登录态。

### 原则 6：静态页面也不是“天然公开的”

只要 APP 处于标准模式，哪怕只是一个静态目录，系统也可以在网关层自动把它变成“登录后才能访问”的页面。

### 原则 7：系统服务的 UI 在网关眼里仍然是 APP

只要你想通过浏览器访问它，它就必须进入 APP / 域名 / 路由 / SSO 这一整套规则体系。

## 16. 仍在演进中的设计点

根据录音，下面这些点仍然属于“方向基本明确，但实现和策略还在演进”：

1. **内核用 JWT helper 与用户态 helper 是否拆分**
2. **CLI / Agent Tool 的权限边界到底落在 Agent 还是 Tool**
3. **非 owner 的 sudo 是否进一步收敛为 `verify-hub` 流程**
4. **多用户共享 APP 在长任务中如何优雅地处理中途 token 续期与身份保持**
5. **BuckyOS App 运行时如何在不破坏现有 Web SDK 语义的前提下增强本地能力**

## 17. 总结

如果只记一件事，可以记住下面这句话：

> **BuckyOS 的分布式权限系统，本质上是“JWT 身份 + 双主体 RBAC + Gateway 流量控制 + 多运行时授权策略”共同组成的一套调用链约束。**

其中：

- **JWT** 负责表达“谁在什么上下文里发起请求”
- **RBAC** 负责判断“这两个主体能否访问这个资源”
- **Gateway** 负责判断“这股流量能否被送到目标位置”
- **不同运行时** 负责决定“这个请求应该拿谁的身份继续往下走”

当你带着这个框架回头看代码时，系统就不再是一堆零散规则，而会变成一张可以逐层 debug 的图。
