# SDK Test环境构造

SDK测试依赖buckyos DV Test环境，应该优先准备好。
## DV test 环境验证
GET http://test.buckyos.io/1.0/identifiers/self 能成功得到ZoneDocument(ZoneConfig)
该环境的用户名是devtest,密码是bucky2025

## AppClient

最简单，直接运行即可.主要appid选择buckycli。
这个appid的权限很大,所以比较适合用来做各种SDK功能的测试。结合devtest本身用户的身份是可以100%访问全部的功能的

## 通过DV环境自带的测试App systest 完成另外两个环境的测试

1）定位运行中的BuckyOS 目录 $BUCKYOS_ROOT (Windows $APPDATA/buckyos, UnixLike:/opt/buckyos/)
   通过得到运行中的cyfs-gateway的进程可以更简单知道
2）阅读代码可知，这个例子非常纯粹:通过ts启动了一个简单的Http服务，然后http服务serve了 ./dist目录

上述结构可以让我们方便的测试SDK。

### Browser
1) 测试代码直接复制到$BUCKYOS_ROOT/bin/buckyos_systest/dist/test.html 
2) 更新测试代码后，可以通过 http://systest.test.buckyos.io/test.html 访问，触发测试代码运行
   如果测试依赖 SDK 写入 `Secure` cookie，应该优先用 https://systest.test.buckyos.io/test.html 做真实浏览器验证，否则浏览器不会保存该 cookie
3）用playwright 或其它测试工具测试
4）从 2026-04-06 起，Browser runtime 的主验证流程应该优先覆盖 `buckyos.login()` 触发的 SSO 跳转链路；直接调用 `verify-hub.login_by_password` 只作为向下兼容验证，不应再作为主流程

### AppService
1) 测试代码直接复制到$BUCKYOS_ROOT/bin/buckyos_systest/main.ts
2) 关闭容器devtest-buckyos_systest，等待node_daemon重新基于新代码运行容器

## 类似Jarvis的本地测试模式

BuckyOS为App运行提供了容器隔离，但这可能会带来一些本地调试的不便。
只要故意不运行Docker服务，就有机会用手工启动的服务来填补位置：

1）有一个$BUCKYOS_ROOT/bin/service_debug.tsx 可以模拟node_daemon启动AppService的流程，并构造正确的环境变量
2）通过service_debug本地运行服务，注意监听10176端口 （这个端口和preinstall的app的顺序有关，DV环境目前基本固定是10176）
3）BuckyOS的网关并不会在意这个服务是谁启动的,hack成功，可以进行本地测试了（访问http://systest.test.buckyos.io/成功)
