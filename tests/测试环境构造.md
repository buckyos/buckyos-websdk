# SDK Test环境构造

SDK测试依赖buckyos DV Test环境，应该优先准备好。
## DV test 环境验证
GET http://test.buckyos.io/1.0/identifiers/self 能成功得到ZoneDocument(ZoneConfig)
该环境的用户名是devtest,密码是bucky2025

## AppClient

最简单，直接运行即可.主要appid选择buckycli。
这个appid的权限很大,所以比较适合用来做各种SDK功能的测试。结合devtest本身用户的身份是可以100%访问全部的功能的

## 通过DV环境自带的测试App systest 完成另外两个环境的测试

1）定位运行中的BuckyOS 目录 $BUCKYOS_ROOT (Windows $APPDATA/buckyos, UnixLike:/opt/buckyos/)
   通过得到运行中的cyfs-gateway的进程可以更简单知道
2）阅读代码可知，这个例子非常纯粹:通过ts启动了一个简单的Http服务，然后http服务serve了 ./dist目录

上述结构可以让我们方便的测试SDK。

### Browser
1) 测试代码直接复制到$BUCKYOS_ROOT/bin/buckyos_systest/dist/test.html 
2) 更新测试代码后，可以通过 http://systest.test.buckyos.io/test.html 访问，触发测试代码运行
   如果测试依赖 SDK 写入 `Secure` cookie，应该优先用 https://systest.test.buckyos.io/test.html 做真实浏览器验证，否则浏览器不会保存该 cookie
3）用playwright 或其它测试工具测试
4）从 2026-04-06 起，Browser runtime 的主验证流程应该优先覆盖 `buckyos.login()` 触发的 SSO 跳转链路；直接调用 `verify-hub.login_by_password` 只作为向下兼容验证，不应再作为主流程

### AppService
1) 测试代码直接复制到$BUCKYOS_ROOT/bin/buckyos_systest/main.ts
2) 关闭容器devtest-buckyos_systest，等待node_daemon重新基于新代码运行容器

## 类似Jarvis的本地Debug模式

BuckyOS为App运行提供了容器隔离，但这可能会带来一些本地调试的不便。
只要故意不运行Docker服务，就有机会用手工启动的服务来填补位置：

1）有一个$BUCKYOS_ROOT/bin/service_debug.tsx 可以模拟node_daemon启动AppService的流程，并构造正确的环境变量
2）通过service_debug本地运行服务，端口以 `$BUCKYOS_ROOT/etc/node_gateway_info.json` 中 `buckyos_systest` 对应的实际端口为准（当前DV环境是10160，该端口和preinstall的app的顺序有关，可能随环境变化）
3）BuckyOS的网关并不会在意这个服务是谁启动的,hack成功，可以进行本地测试了（访问http://systest.test.buckyos.io/成功)

现在 `websdk` 仓库里也保存了一份本地副本和启动脚本：

- `tests/scripts/service_debug.tsx`
- `tests/scripts/debug_systest.sh`
- `tests/app-service/systest/main.ts`
- `tests/app-service/systest/deno.json`

### WebSDK里的最小 AppService 本地 Debug 用例

仓库里现在提供了一个可直接本地运行的 AppService Debug 用例：

- `examples/app_service_local_debug.ts`

它会：

1. 读取 `service_debug.tsx` 注入的 `app_instance_config`
2. 读取 `service_debug.tsx` 注入的 `OWNER_APP_TOKEN`
3. 读取 `service_debug.tsx` 注入的 `BUCKYOS_HOST_GATEWAY`
4. 用 `RuntimeType.AppService` 初始化 SDK 并执行 `buckyos.login()`
5. 在本地启动一个 HTTP 服务，默认监听 `10176`
6. 暴露几个方便调试的接口：
   - `GET /`
   - `GET /healthz`
   - `GET /debug/runtime`
   - `GET /debug/settings?name=foo.bar`
   - `POST /debug/settings`
   - `GET /debug/system-config?key=boot/config`

推荐启动方式：

```bash
# 推荐，直接使用 websdk 仓库里的本地副本
# 端口不用手动指定，脚本会自动从 node_gateway_info.json 解析 buckyos_systest 的实际端口
./tests/scripts/debug_systest.sh

# 或
pnpm run debug:systest
```

如果目标是自动完成全部 AppService 相关测试，而不是只拉起调试服务，可以直接运行：

```bash
# 前提：docker service 已经停掉
./tests/scripts/test_app_service_debug.sh

# 或
pnpm run test:app-service:debug
```

这个脚本会自动完成：

1. 构建 `websdk`
2. 用仓库内的 `scripts/service_debug.tsx` 拉起 `buckyos_systest`
3. 等待 `/sdk/appservice/healthz` 就绪
4. 自动生成给 Jest 使用的 AppService env
5. 跑完 `tests/app-service`
6. 再做一轮网关 smoke check

这个示例不再自己生成 token。
原因是 `service_debug.tsx` 本身就是第三方开发者依赖的调试入口，测试链路应该直接覆盖它。
`debug_systest.sh` 会优先使用 `websdk/scripts/service_debug.tsx`，并自动把 `tests/app-service/systest` 下的入口文件同步到 `/opt/buckyos/bin/buckyos_systest`。

如果环境变量没有注入完整，示例会直接报错，提示缺少：

- `app_instance_config`
- `OWNER_APP_TOKEN`
- `BUCKYOS_HOST_GATEWAY`

调试成功后，除了本地端口，也可以继续用网关入口验证：

```bash
# 端口以 node_gateway_info.json 中的实际值为准（当前是 10160）
curl http://127.0.0.1:10160/healthz
curl http://127.0.0.1:10160/debug/system-config?key=boot/config
curl http://systest.test.buckyos.io/
```
