# 测试用例组织

## 总体设计

1）不同的 Runtime 有不同的测试环境初始化部分
2）初始化成功后，运行使用实现的 xxxServiceClient 的测试（可复用共享套件）
3）调用该 Runtime 的特有测试

所有测试可以一次全部运行进行验证（要求 DV 环境已经存在）。
浏览器环境下要求的 SSO 登录，用 playwright 模拟点击完成。

## 各种 ServiceClient （对应的真实 rust sdk 在 ../../buckyos/src/kernel/buckyos-api/src/）
- systemConfigClient
- verifyHubClient : Runtime 在初始化的时候已经登录过了，无需额外构造测试
- aiccClient
- control_panel : 功能最大的一个分支，但一般服务不会依赖该组件，可以先跳过。由 buckycli 工具的测试进行覆盖
- kevent_client : 先跳过，需要设计符合 ts 的 kevent 整合模式
- msg_queue : 要补充实现，这通常是一个 service_only 的 client
- msg_center_client : 要补充实现
- opendan_client : OpenDan 的 API 已经大改，旧的 `OpenDanClient` 已从 SDK 里彻底删除，等新的 API 稳定后再补回
- repo_client : 要补实现
- scheduler_client : 跳过，这个内核服务不会有 ts 代码有访问需求
- sn_client : 跳过，这个是 zone 外服务，不在 sdk 中
- task_mgr

## 测试层级与对应位置

对齐 WEB_SDK_TEST_PLAN.md 中定义的三层：

| 层级 | 目标 | 是否需要 DV 环境 | 存放位置 |
|------|------|------------------|----------|
| Layer 1/2 ServiceClient 单元+Mock 测试 | 每个 `xxxServiceClient` 的 RPC 请求/响应形状、缓存语义（当前以 mock fetcher 驱动，仅作过渡形态，最终依靠 Layer 3 真实环境验证） | 否 | `tests/*.test.ts`（平铺：`krpc_client`、`verify_hub_client`、`system_config_client`、`task_mgr_client`） |
| Layer 1 Runtime 单元测试 | Browser/AppClient/AppService 各自的初始化、存储、URL 解析等本地行为 | 否 | `tests/browser/*.test.ts`、`tests/app-client/runtime.unit.test.ts`、`tests/app-service/runtime.appservice.test.ts` |
| Layer 3 环境回归集成测试 | 对接真实 `test.buckyos.io`，通过共享 ServiceClient 套件统一验收 | 是（由 `BUCKYOS_RUN_INTEGRATION_TESTS=1` 打开） | `tests/browser/real-browser/`、`tests/app-client/integration/`、`tests/app-service/systest/` |

## 三阶段结构（Layer 3 集成测试）

所有 Runtime 集成测试都按照以下三阶段组织：

- **阶段 1 · Runtime 初始化（`beforeAll`）**
  - Browser：只通过真实浏览器 + playwright 完成。`test.html` 内 `initBuckyOS`，由 playwright 驱动 SSO 点击获取 session token（jsdom + `loginByPassword` 的路径已废弃，因为拿不到可用的 session token，无法驱动后续 ServiceClient 测试）
  - AppClient：加载本地私钥目录 → `initBuckyOS` → 生成本地 JWT → `login()`
  - AppService：从 `app_instance_config` 与 token env 启动 → `initBuckyOS` → `login()`（`systest/main.ts`）
- **阶段 2 · 共享 ServiceClient 套件**
  - 通过 `defineSharedServiceClientSuite` 注入已初始化的 `buckyos` 单例
  - 覆盖：`SystemConfigClient` 读 `boot/config`、用户命名空间写回、`getAppSetting/setAppSetting` 往返、`TaskManagerClient` 创建/更新/查询/删除
  - 通过 `skipSettings`、`skipTaskManager` 开关适配不同 Runtime 的权限与能力
- **阶段 3 · Runtime 特有断言**
  - Browser：`localStorage`/cookie 按 appId 分桶
  - AppClient：本地 JWT 中的 `appid`/`sub` claim、`user_id` 与 subject 一致
  - AppService（systest）：`/sdk/appservice/runtime`、`/settings`、`/system-config` 路由返回的字段

## 共享套件（`helpers/service_client_suite.ts`）

用途：统一阶段 2 的 ServiceClient 断言，避免每个 Runtime 重复实现。

调用方契约：
- `getSdk()` 返回已初始化 + 登录完成的 `buckyos` 单例
- `getAppId()` / `getUserId()` 提供测试数据的命名空间
- `skipTaskManager` / `skipSettings` 用来跳过当前 Runtime 不适用的用例

命名空间约定：
- SystemConfig 写入键：`users/${userId}/test_websdk/${appId}/${timestamp}`
- Settings 路径：`test_settings.websdk_${timestamp}`
- TaskManager 任务名：`test-websdk-${timestamp}`

## 测试环境开关

全部位于 `helpers/test_env.ts`：

- `BUCKYOS_RUN_INTEGRATION_TESTS=1` — 打开 Layer 3 集成测试（`describeIntegration`）
- `BUCKYOS_TEST_ZONE_HOST` / `BUCKYOS_TEST_APP_ID` / `BUCKYOS_TEST_USERNAME` / `BUCKYOS_TEST_PASSWORD` — 登录凭证与目标 zone
- `BUCKYOS_TEST_APP_CLIENT_DIR` — AppClient 私钥搜索目录
- `BUCKYOS_TEST_VERIFY_HUB_URL` / `BUCKYOS_TEST_SYSTEM_CONFIG_URL` — 覆盖服务地址
- `BUCKYOS_HOST_GATEWAY` — AppService 使用的 host gateway
- `app_instance_config` + `<OWNER>_<APP>_TOKEN` / `<APP>_TOKEN` — AppService 启动输入

对于 `test.buckyos.io` 及其子域，`configureInsecureTlsIfNeeded` / `installInsecureNodeFetchIfNeeded` 会自动关闭证书校验，便于在 DV 环境下运行。
