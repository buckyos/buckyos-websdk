# 测试用例组织

## 总体设计

**每个可被 jest 驱动的 Runtime 都跑完整的三阶段流程：**

1）Runtime 初始化 + `login()`（拿到可用 session token）
2）所有通用的 ServiceClient 测试（共享套件统一实现）
3）该 Runtime 的特有断言

所有测试可以一次全部运行进行验证（要求 DV 环境已经存在）。

> **jsdom 已彻底移除。** jsdom 既拿不到可用的 session token，也没办法驱动真实的 SSO 跳转，对 Browser runtime 的覆盖没有意义。Jest 的默认测试环境现在是 `node`，Browser runtime 的真实覆盖完全交给 `tests/browser/real-browser/playwright.spec.js` 在真实浏览器里完成。

> **AppService 的 phase 1 走 systest 槽位。** 以前 jest 靠 `process.env.app_instance_config = ...` + `<OWNER>_<APP>_TOKEN = ...`，再在 jest 进程内 `initBuckyOS(AppService)` 的做法在真实 DV 环境下不可用——这些 token 是 node-daemon 临时签发的，jest 无法凭空合成。现在 AppService 集成测试的 phase 1 改为：由 `tests/scripts/debug_systest.sh`（= `service_debug.tsx` + `tests/app-service/systest/main.ts`）起一个**真实 AppService 进程**，jest 只通过 HTTP 驱动它。共享 ServiceClient 套件（phase 2）直接跑在 systest 进程里（对应 `POST /sdk/appservice/selftest` 端点），jest 只拿结果断言。AppClient 没有这个限制，仍然维持 jest 进程内 init+login 的传统做法。

## 各种 ServiceClient （对应的真实 rust sdk 在 ../../buckyos/src/kernel/buckyos-api/src/）

- systemConfigClient : 参考rust完善接口，补充测试
- verifyHubClient : Runtime 在初始化的时候已经登录过了，无需额外构造测试
- aiccClient
- control_panel : 功能最大的一个分支，但一般服务不会依赖该组件，可以先跳过。由 buckycli 工具的测试进行覆盖
- kevent_client : 先跳过，需要设计符合 ts 的 kevent 整合模式
- msg_queue : 要补充实现，这通常是一个 service_only 的 client
- msg_center_client : 要补充实现
- opendan_client : OpenDan 的 API 已经大改，旧的 `OpenDanClient` 已从 SDK 里彻底删除，等新的 API 稳定后再补回
- repo_client : 要补实现
- scheduler_client : 跳过，这个内核服务不会有 ts 代码有访问需求
- sn_client : 跳过，这个是 zone 外服务，不在 sdk 中
- task_mgr

## 测试层级

| 层级 | 目标 | 需要 DV 环境 | 存放位置 |
|------|------|--------------|----------|
| Layer 1/2 ServiceClient 单元+Mock 测试 | 每个 `xxxServiceClient` 的 RPC 请求/响应形状、缓存语义（当前以 mock fetcher 驱动，过渡形态，最终依靠 Layer 3 真实环境验证） | 否 | `tests/*.test.ts`（平铺：`krpc_client`、`verify_hub_client`、`system_config_client`、`task_mgr_client`） |
| Layer 1 Runtime 单元测试 | AppClient / AppService 各自的初始化、URL 解析等本地行为 | 否 | `tests/app-client/runtime.unit.test.ts`、`tests/app-service/runtime.appservice.test.ts`、`tests/browser/auth_client.node.test.ts` |
| Layer 3 环境回归集成测试 | 对接真实 `test.buckyos.io`，每个可测试的 Runtime 都走完整三阶段 | 是（由 `BUCKYOS_RUN_INTEGRATION_TESTS=1` 打开） | `tests/app-client/integration/`、`tests/app-service/integration/`（通过 systest 槽位）、`tests/app-service/systest/`、`tests/browser/real-browser/`（playwright） |

## 三阶段结构（Layer 3 集成测试）

Jest 能自动驱动的两个 Runtime 都严格按三阶段组织，只是 AppService 的 phase 1 / phase 2 实际执行地点被搬到了 systest 进程。

### 阶段 1 · Runtime 初始化 + login（`beforeAll`）

- **AppClient**（`tests/app-client/integration/app_client_test.ts`）
  - 从 `BUCKYOS_TEST_APP_CLIENT_DIR` 等路径加载本地私钥 → `buckyos.initBuckyOS(RuntimeType.AppClient, ...)` → `buckyos.login()` → 运行时生成本地 JWT，再向 verify-hub 换取 session token。整个过程都在 jest 进程内完成。
- **AppService**（`tests/app-service/integration/app_service_test.ts`，**通过 systest 槽位**）
  - 真正的 `initBuckyOS(RuntimeType.AppService)` + `login()` 发生在 `tests/app-service/systest/main.ts` 进程里。systest 进程由 `tests/scripts/debug_systest.sh` → `service_debug.tsx` 启动，node-daemon 风格的 env（`app_instance_config`、`BUCKYOS_HOST_GATEWAY`、`<OWNER>_<APP>_TOKEN`）由 service_debug.tsx 注入。
  - jest 侧 `beforeAll` 只做一件事：`GET ${BUCKYOS_TEST_SYSTEST_URL}/sdk/appservice/healthz` + `/runtime` 确认 systest 已经 login 完成，然后把 `/runtime` 返回的元数据存起来，留给 phase 3 断言。

### 阶段 2 · 共享 ServiceClient 套件

共享套件的 **case 列表** 是唯一事实来源，包含：

- `SystemConfigClient.get('boot/config')` — 读 zone 级系统配置
- `SystemConfigClient` 在 `users/${userId}/apps/${appId}/info` 下写入并读回（值里塞 `ts: ${timestamp}` 保证每次都是新内容；这是 `app` 角色 RBAC 策略允许的写入路径，AppClient/AppService 都能跑）
- `getAppSetting` / `setAppSetting` 在 `test_settings.websdk_${timestamp}` 下的往返
- `TaskManagerClient` 在 `test-websdk-${timestamp}` 下的完整生命周期：create → update progress/status → get/list → delete

两个实现入口（同一套 case，两套宿主）：

- **AppClient**：jest 进程内调用 `defineSharedServiceClientSuite({ getSdk, getAppId, getUserId })`，直接在 jest 里对已 login 的 `buckyos` 单例跑断言（`tests/helpers/service_client_suite.ts`）。
- **AppService**：case 的真实执行搬到 `tests/app-service/systest/main.ts` 里的 `runSharedServiceClientSelftest()`，由 `POST /sdk/appservice/selftest` 端点暴露。jest 侧的 AppService 集成测试只是 `POST` 这个端点，然后把返回的 `{ ok, results[] }` 逐项断言。两份实现的 case 命名和覆盖点保持一一对应，避免漂移。

### 阶段 3 · Runtime 特有断言

- **AppClient**：本地 JWT 中的 `appid` claim 等于 `BUCKYOS_TEST_APP_ID`，`user_id` 与 token subject 一致
- **AppService**（通过读 systest 的 `/sdk/appservice/runtime` 返回值断言）：
  - `mode === 'app-service-local-debug'`
  - `serviceUrls.verifyHub` / `taskManager` / `systemConfig` 都包含注入的 `BUCKYOS_HOST_GATEWAY`
  - `expectedTokenEnvKey` 符合 Rust 风格的 `<OWNER>_<APP>_TOKEN` 命名约定
  - `accountInfo.userId` 非空，表示 systest 侧的 login 真的成功了

## Browser runtime 如何覆盖

Browser runtime 的原生 login 路径是 SSO 跳转（`sys.${zoneHost}/login`）。这个流程无法在 jest/jsdom 里复现——之前尝试过的 `loginByPassword` 拿不到可用于后续 ServiceClient 调用的 token，而 "在 jsdom 里 bootstrap 一个 token 再塞给 Browser runtime" 的变通方案等于没在测 Browser runtime 的真实 login 路径。所以现在 Browser runtime **完全不在 jest 里测**，只通过真实浏览器 + playwright 覆盖：

- `tests/browser/real-browser/test.html` 加载打包好的 SDK，初始化 Browser runtime。
- `tests/browser/real-browser/playwright.spec.js` 用真实 Chromium 打开该页面，验证 `#login-button` 触发的 SSO URL 等于 SDK 生成的 URL。
- 如果后续要让真实浏览器也跑共享套件，需要在 `test.html` 里内联浏览器版的 ServiceClient 断言（import 打包好的 `dist/browser.mjs`），让 playwright 在 SSO 回跳后读取页面里的 `#result`。目前这部分是已知缺口。

`tests/browser/` 下的 jest 文件只剩 `auth_client.node.test.ts`，它验证 `AuthClient` 在 node 环境下会拒绝构造——属于一个和 runtime 初始化/SSO 流程相关的静态保护性回归，和 Browser runtime 的实际运行无关，可以在 node 环境下跑。

## 共享套件的两套实现

| 实现文件 | 宿主 | 谁在用 | 触发方式 |
|----------|------|--------|----------|
| `tests/helpers/service_client_suite.ts`（`defineSharedServiceClientSuite`） | jest 进程（真实 SDK 单例已在进程内 login） | AppClient 集成测试 | 直接 import 调用，注册 `describe/it` |
| `tests/app-service/systest/main.ts`（`runSharedServiceClientSelftest`） | systest 进程（真实 AppService 已在进程内 login） | AppService 集成测试 | `POST /sdk/appservice/selftest`，jest 拿 JSON 结果做断言 |

两份实现必须保持一一对应：增减 case 时两边都要改。可以考虑未来把 case 定义提炼成一个"纯描述列表"然后两个 runner 各自消费，不过现在两份都很短，就先这样维护。

命名空间约定（两边共用，避免污染 DV 环境）：
- SystemConfig 写入键：`users/${userId}/apps/${appId}/info`（值里嵌 `ts: ${timestamp}` 保证每次写入都是新内容）。这是 `app` 角色 RBAC 策略 (`/config/users/*/apps/{app}/info`) 允许写入的固定 key，AppClient/AppService 都能跑。
- Settings 路径：`test_settings.websdk_${timestamp}`
- TaskManager 任务名：`test-websdk-${timestamp}`

## 运行 AppService 集成测试的流程

```bash
# 一条命令（已经装好 service_debug.tsx + buckyos 真实环境）：
./tests/scripts/test_app_service_debug.sh devtest --port 10176

# 或者手动分两步，便于调试 systest 本身：
./tests/scripts/debug_systest.sh devtest --port 10176   # 起 systest（前台），注入真实 token env
# 另开一个窗口：
BUCKYOS_RUN_INTEGRATION_TESTS=1 \
  BUCKYOS_TEST_SYSTEST_URL=http://127.0.0.1:10176 \
  pnpm exec jest --runInBand tests/app-service
```

## 测试环境开关

全部位于 `helpers/test_env.ts`：

- `BUCKYOS_RUN_INTEGRATION_TESTS=1` — 打开 Layer 3 集成测试
- `BUCKYOS_TEST_ZONE_HOST` / `BUCKYOS_TEST_APP_ID` — 目标 zone 与 appId
- `BUCKYOS_TEST_APP_CLIENT_DIR` — AppClient 私钥搜索目录
- `BUCKYOS_TEST_VERIFY_HUB_URL` / `BUCKYOS_TEST_SYSTEM_CONFIG_URL` — 覆盖服务地址
- `BUCKYOS_TEST_OWNER_USER_ID` — AppService 集成测试用来校验 systest 返回的 identity
- `BUCKYOS_TEST_SYSTEST_URL` — AppService 集成测试指向的 systest 基础 URL（默认 `http://127.0.0.1:10176`）
- `BUCKYOS_TEST_HOST_GATEWAY` / `BUCKYOS_HOST_GATEWAY` — 期望的 AppService host gateway，用于断言 systest `/runtime` 返回的 `hostGateway` 字段

> 注意：原来的 `BUCKYOS_TEST_APP_SERVICE_TOKEN` / `BUCKYOS_TEST_APP_INSTANCE_CONFIG` 已经废弃。真实 token 和 `app_instance_config` 现在由 `service_debug.tsx` 直接注入到 systest 进程，jest 不再需要自己合成这些。

对于 `test.buckyos.io` 及其子域，`configureInsecureTlsIfNeeded` / `installInsecureNodeFetchIfNeeded` 会自动关闭证书校验并把相对 URL 重写到 zoneHost，便于在 DV 环境下运行——这些现在只给 AppClient 和 systest 进程使用，jest 侧的 AppService 集成测试只发本地 HTTP 到 systest，用不到。
